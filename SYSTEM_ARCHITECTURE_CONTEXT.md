# [MASTER_CONTEXT] 거대 자율 수행 에이전트 시스템 아키텍처 명세서

## 1. 프로젝트 비전 및 엔지니어링 철학
본 프로젝트의 최종 목표는 단순한 챗봇이나 프롬프트 래퍼(Wrapper)를 구축하는 것이 아니다. 환경을 인지하고 다단계 추론을 통해 스스로 목표를 달성하는 **'독립적인 디지털 노동자'**를 구축하는 것이다.
*   **하네스 엔지니어링 (Harness Engineering):** LLM은 본질적으로 비결정론적(Probabilistic)이고 확률적인 엔진이다. 당신(AI 개발 에이전트)은 이 확률적 엔진이 발산(Divergence)하거나 환각(Hallucination)에 빠지지 않도록 물리적으로 통제하는 결정론적(Deterministic) 제어 평면인 **'하네스(Harness)' 인프라**를 최우선으로 설계해야 한다.

## 2. 코어 아키텍처 원칙 (GEODE 참조 모델)
당신이 코드를 작성할 때 반드시 기저에 깔려 있어야 할 아키텍처 원칙이다.

### 2.1. 5계층 검증 파이프라인 (5-Layer Verification)
에이전트의 모든 출력(코드, 계획 등)은 다음의 결정론적 관문을 통과해야만 한다.
1. **원자적 XML 매칭:** 출력은 사전 정의된 XML 앵커 구조(`<think>`, `<action>` 등)를 완벽히 준수해야 한다.
2. **구문 빗자루 (Syntactic Broom):** 작성된 코드는 샌드박스 실행 전 린터(Linter)와 포매터의 자동 검증을 거쳐 구문 오류를 청소해야 한다.
3. **심볼릭 리플 엔진:** 단일 모듈의 변경이 시스템 하위 의존성에 미치는 파급 효과를 자동 추적한다.
4. **컨텍스트 인지 자가 치유 (Self-Healing):** 실행 중 오류 발생 시, 사람 개입 없이 원본 목표와 에러 트레이스백을 피드백으로 삼아 스스로 코드를 교정해야 한다.
5. **런타임 스모크 테스트:** 제한된 시간 내에 런타임 에러 없이 동작함을 물리적으로 증명해야 한다.

### 2.2. 로우레벨 인프라 및 통신 제어
*   **결정론적 샌드박싱:** 에이전트가 생성한 코드는 절대 호스트 OS에서 직접 실행될 수 없다. 호스트 OS의 프로세스 제한과 충돌을 막기 위해 반드시 권한이 극도로 제한된 컨테이너(Docker/VM) 내부에서만 실행되어야 한다.
*   **통신 및 동시성:** 메인 오케스트레이터와 에이전트 간의 프로세스 통신은 오버헤드가 적은 유닉스 도메인 소켓(Unix Domain Sockets)을 지향한다. 또한, 세션 고유 키 기반의 세마포어(SessionLane)로 병렬 충돌을 방지하고, API 통신 시 난수가 포함된 지터(Jitter) 결합 백오프를 적용한다.

### 2.3. 절대 방어선 (Safety Guardrails)
*   **예산 게이트 및 회로 차단기:** 에이전트별 토큰 사용량/비용의 하드 리밋(Hard Limit)을 설정한다. 예산 100% 도달 즉시 어플리케이션 계층 이전에 네트워크 요청을 물리적으로 차단하고 강제 정지시킨다.
*   **의미론적 무한 루프 감지:** 에이전트의 추론 흔적을 임베딩하여 코사인 유사도를 계산한다. 의미론적으로 동일한 실패 궤적을 반복하면 즉시 실행을 중지(Human-in-the-loop)한다.

### 2.4. 다중 에이전트 오케스트레이션 (Science of Scaling)
*   확장을 위해 무분별하게 에이전트의 수를 늘리는 병렬 배치를 절대 금지한다 (오류 17.2배 증폭 위험).
*   반드시 중앙 오케스트레이터가 품질 게이트(Quality Gate) 역할을 수행하며 하위 에이전트를 통제하는 구조를 취한다.
*   명시적 상태(State) 기반 워크플로우 제어를 위해 코어 프레임워크로 **`LangGraph`**를 채택한다.
